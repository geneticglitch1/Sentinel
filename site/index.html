for recruiters · what this demonstrates

A full AI agent system, end to end

Not a wrapper around a chatbot. Sentinel is a real agentic system: a typed tool layer, an MCP server, a Claude Agent SDK loop, a service API, a CLI, and a dashboard — designed, built, and tested as one. Here's the range it shows.

◆ Agentic AI, properly

An MCP server (Claude Desktop + Claude Code) and a Claude Agent SDK loop share one typed tool layer — natural language → planned tool calls → execution. Opus 4.8 with adaptive thinking. Translates "deploy postgres, expose 5432" into a real command.

◆ Real infrastructure automation

Drives Proxmox, an OPNsense firewall, and a Docker host over SSH, REST, and the Proxmox API — VM lifecycle, container deploys, firewall rules, Suricata IPS, CrowdSec. Production-style homelab, not a toy.

◆ Safety & reliability engineering

Dry-run-by-default, destructive-op refusal, a SQLite audit trail, and supported-only apply paths — designed out of a real lockout incident. The judgment a recruiter actually wants to see.

◆ Full-stack delivery

Python backend, FastAPI service, Typer CLI, a Next.js dashboard, Docker packaging, and a real test suite. Shipped as a coherent product, documented, and demoable.

the flagship

One sentence → a deploy

Plain English becomes a real docker run. It plans first — shows you the exact command — then waits for the confirm. The demo above runs this end to end.

You say

"run postgres:16 on the docker host, 2 GB RAM, expose 5432, volume pgdata"

Sentinel plans

docker run -d --name pg --memory 2g -p 5432:5432 -v pgdata:/var/lib/postgresql/data postgres:16

why it's trustworthy

Safe by default

A control plane can lock you out via the firewall or nuke a VM. So the rule is strict, and it lives in one place: nothing changes the homelab without passing the gate.

  • Dry-run first.Every mutating tool returns a plan and touches nothing — unless you pass confirm=true.
  • Destructive ops refuse.Stop a VM, drop a firewall rule, rm a container? Blocked without an explicit confirm.
  • Everything audited.Planned or applied, every call lands in a SQLite log the dashboard reads.
  • Supported paths only.OPNsense changes go through configctl / REST — never raw PHP — and back up config.xml first.

a real lesson

Hardening the firewall by hand once, I stopped sshd mid-change and locked myself out. Recoverable — but never again. That incident is the reason the OPNsense provider is the way it is: backup, supported apply path, verify the listener is back. The mistake is designed out.

Providers do the raw I/O and stay honest about side effects. The tool layer owns the dry-run / confirm gate and the audit log. That split is the whole reason the safety story holds.
